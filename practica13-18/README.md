# PWA Notes

PWA-приложение для заметок, собранное по методическим материалам практик 13–18 по дисциплине «Фронтенд и бэкенд разработка».

Приложение работает по `HTTPS`, поддерживает офлайн-режим, push-уведомления, отложенные напоминания и обновление интерфейса через `WebSocket`.

## Возможности

- добавление обычных заметок с мгновенным push-уведомлением;
- создание заметок с отложенным напоминанием по дате и времени;
- удаление заметок и автоматическая отмена связанного напоминания на сервере;
- полноценная работа в офлайн-режиме благодаря `Service Worker` и кешированию ресурсов;
- установка приложения на устройство через `Web App Manifest`;
- просмотр информации о локальном `HTTPS`-сертификате на вкладке «О приложении»;
- синхронное обновление интерфейса между вкладками через `Socket.IO`;
- действие «Отложить на 5 минут» прямо из push-уведомления.

## Стек

| Технология | Назначение |
|---|---|
| `HTML`, `CSS`, `JavaScript` | Клиентская часть (SPA без фреймворков) |
| `Express` | HTTP/HTTPS-сервер, раздача статики и REST API |
| `Socket.IO` | Двусторонняя связь в реальном времени (WebSocket) |

## Соответствие практикам

### Практика 13 — Офлайн-режим и Service Worker

В рамках этой практики было реализовано ядро офлайн-функциональности приложения — `Service Worker`.
Файл `sw.js` регистрируется при загрузке страницы из `app.js` через вызов `navigator.serviceWorker.register('/sw.js')`.

**Стратегия кеширования.** Используются два именованных кеша:

- `notes-cache-v3` — статический кеш для App Shell (HTML, CSS, JS, иконки, `manifest.json`, клиентская библиотека `socket.io`). Все ресурсы из массива `ASSETS` предварительно загружаются на этапе `install`, что гарантирует их доступность при полном отключении сети.
- `dynamic-content-v2` — динамический кеш для HTML-фрагментов страниц `home.html` и `about.html`. Эти страницы предзагружаются при установке SW, а затем обновляются по стратегии **Network First**: если сеть доступна, берётся свежий ответ (и сохраняется в кеш), иначе возвращается закешированная версия.

**Жизненный цикл.** При событии `activate` удаляются все кеши, которые не совпадают с текущими именами (`CACHE_NAME`, `DYNAMIC_CACHE_NAME`). Это обеспечивает автоматическую очистку устаревших ресурсов при обновлении версии SW. Вызов `self.skipWaiting()` и `self.clients.claim()` гарантирует, что новый SW активируется немедленно, без ожидания закрытия старых вкладок.

**Перехват запросов.** Обработчик `fetch` фильтрует только GET-запросы к текущему origin, пропуская транспортные запросы Socket.IO. Для маршрутов `/content/*` применяется стратегия Network First с fallback на кеш, а для прочих ресурсов — Cache First, что минимизирует сетевые запросы для неменяющихся файлов.

### Практика 14 — Web App Manifest и иконки

Практика посвящена превращению обычного веб-сайта в устанавливаемое PWA-приложение.

**Web App Manifest** (`manifest.json`) содержит всю необходимую метаинформацию для установки:

- `name` / `short_name` — полное и сокращённое название приложения, отображаемые на сплэш-скрине и в лаунчере устройства;
- `start_url: "/"` — URL, с которого запускается приложение после установки;
- `display: "standalone"` — режим отображения без адресной строки браузера, имитирующий нативное приложение;
- `background_color` / `theme_color` — цвета загрузочного экрана и системного UI (статус-бар на Android);
- `orientation: "portrait-primary"` — фиксированная портретная ориентация.

**Набор иконок.** Каталог `icons/` содержит иконки в разрешениях от 16×16 до 512×512 пикселей, каждая прописана в секции `icons` манифеста. Иконка 512×512 имеет `purpose: "maskable any"`, что позволяет операционной системе обрезать её под адаптивную форму (круг, скруглённый квадрат). В `index.html` подключены `<link rel="manifest">`, `<link rel="apple-touch-icon">` для iOS и `<meta name="theme-color">` для корректного отображения на всех платформах.

**Мета-теги установки.** Для полной совместимости с Safari/iOS в `<head>` добавлены `<meta name="mobile-web-app-capable" content="yes">` и `<meta name="apple-mobile-web-app-status-bar-style" content="black-translucent">`, обеспечивающие полноэкранный режим и полупрозрачный статус-бар при запуске с домашнего экрана.

### Практика 15 — HTTPS и App Shell

Практика охватывает два направления: обеспечение безопасного HTTPS-соединения и архитектурный паттерн App Shell.

**HTTPS.** Сервер поддерживает два сценария получения TLS-сертификата:

1. **mkcert** — если в корне проекта присутствуют файлы `localhost.pem` и `localhost-key.pem` (сгенерированные утилитой `mkcert`), сервер использует их. Такой сертификат доверенный на уровне системы и не вызывает предупреждения в браузере.
2. **selfsigned** — если файлов mkcert нет, сервер автоматически генерирует самоподписанный сертификат (SHA-256, RSA 2048 бит) через библиотеку `selfsigned` и сохраняет его в `cert.json`. Сертификат включает расширения `subjectAltName` (localhost, 127.0.0.1, ::1), `basicConstraints`, `keyUsage` и `extKeyUsage`.

Информация о текущем сертификате парсится через Node.js `X509Certificate` и доступна через API-эндпоинт `GET /api/https/certificate`. На вкладке «О приложении» отображаются: subject, issuer, дата начала/окончания действия, SHA-256 отпечаток и полный PEM сертификата в раскрывающемся блоке `<details>`.

**App Shell.** Архитектура приложения построена по паттерну App Shell: файл `index.html` содержит только фиксированный каркас — навигационные табы («Главная» / «О приложении»), кнопки push-уведомлений и контейнер `<main id="app-content">`. Содержимое страниц (`content/home.html`, `content/about.html`) загружается динамически через `fetch` и вставляется в контейнер. Благодаря прекешированию App Shell в Service Worker, каркас отображается мгновенно даже без сети, а контент подгружается из динамического кеша.

### Практика 16 — WebSocket и push-уведомления

Практика посвящена реализации двусторонней связи между клиентом и сервером в реальном времени, а также интеграции с Web Push API.

**WebSocket (Socket.IO).** На сервере создаётся инстанс `Server` из пакета `socket.io`, привязанный к существующему HTTPS-серверу. Клиент подключается автоматически через скрипт `socket.io.js`. Реализованы два события:

- `newTask` — клиент отправляет текст новой заметки; сервер транслирует событие `taskAdded` всем подключённым клиентам (broadcast), после чего каждый клиент показывает toast-уведомление. Одновременно сервер вызывает `sendPushToSubscriptions` для рассылки push-уведомления.
- `newReminder` — клиент отправляет текст и время напоминания; сервер планирует отложенную push-рассылку через `setTimeout` и сохраняет таймер в `Map` для возможности отмены.

**Push-уведомления** работают по стандартному протоколу Web Push с VAPID-аутентификацией:

1. При старте сервер загружает или генерирует пару VAPID-ключей (`web-push.generateVAPIDKeys()`) и сохраняет их в `vapid.json`.
2. Клиент получает публичный ключ через `GET /vapidPublicKey`, конвертирует его в `Uint8Array` и подписывается на push через `PushManager.subscribe()`.
3. Подписка (`PushSubscription`) отправляется на сервер через `POST /subscribe` и хранится в массиве `subscriptions`.
4. При создании заметки сервер рассылает push через `webPush.sendNotification()`. Если подписка устарела (HTTP 404/410), она автоматически удаляется из массива.

На клиенте реализованы кнопки «Включить уведомления» и «Отключить уведомления» с проверкой `Notification.permission` и запросом разрешения через `Notification.requestPermission()`. Отписка выполняется через `POST /unsubscribe` и `subscription.unsubscribe()`.

### Практика 17 — Напоминания и отложенные push-уведомления

Практика расширяет систему push-уведомлений возможностью планирования отложенных напоминаний и интерактивными действиями в уведомлениях.

**Отложенные напоминания.** На главной странице расположена вторая форма с полями «текст» и `<input type="datetime-local">`. При отправке формы клиент:

1. Валидирует, что выбранная дата находится в будущем.
2. Сохраняет заметку в `localStorage` с полем `reminder` (timestamp).
3. Отправляет событие `newReminder` через Socket.IO на сервер с параметрами `id`, `text` и `reminderTime`.

Сервер вычисляет задержку `delay = reminderTime - Date.now()` и, если она положительная, создаёт `setTimeout`. Таймер сохраняется в `Map<id, { timeoutId, text, reminderTime }>`. По истечении задержки вызывается `sendPushToSubscriptions` с заголовком «!!! Напоминание» и `reminderId` в payload. Если для того же `id` уже существовал запланированный таймер, он предварительно отменяется через `clearTimeout`.

**Действие «Отложить на 5 минут» (snooze).** В Service Worker обработчик `push` проверяет наличие `reminderId` в данных push-события. Если ID присутствует, к уведомлению добавляется кнопка-действие `{ action: 'snooze', title: 'Отложить на 5 минут' }`. При нажатии на эту кнопку обработчик `notificationclick` отправляет запрос `POST /snooze?reminderId=<id>` на сервер. Сервер отменяет текущий таймер и планирует новый с задержкой 5 минут (`5 × 60 × 1000 мс`), обновляя запись в `Map`.

**Удаление напоминания.** При удалении заметки из списка клиент проверяет наличие поля `reminder` и отправляет `DELETE /reminders/:id` на сервер. Сервер останавливает таймер через `clearTimeout` и удаляет запись из `Map`, предотвращая отправку push для несуществующей заметки.

**Нажатие на уведомление.** Если пользователь нажимает на тело уведомления (а не на действие snooze), обработчик `notificationclick` ищет открытую вкладку приложения через `clients.matchAll()` и фокусирует её. Если ни одна вкладка не найдена, открывается новое окно через `clients.openWindow('/')`.

1. Установить зависимости:

```bash
npm install
```

2. Запустить сервер:

```bash
npm start
```

3. Открыть приложение в браузере:

```text
https://localhost:3000
```

## HTTPS

- Если в корне проекта есть `localhost.pem` и `localhost-key.pem`, сервер использует их (рекомендуется `mkcert` для доверенного сертификата).
- Если этих файлов нет, сервер автоматически создаёт самоподписанный сертификат и хранит его в `cert.json`.
- Информация о сертификате доступна на вкладке «О приложении» и через `GET /api/https/certificate`.

## API-эндпоинты

| Метод | URL | Описание |
|---|---|---|
| `GET` | `/vapidPublicKey` | Вернуть публичный VAPID-ключ (text/plain) |
| `GET` | `/api/status` | Информация о сервере: подписки, таймеры, сертификат |
| `GET` | `/api/https/certificate` | JSON с данными HTTPS-сертификата |
| `POST` | `/subscribe` | Сохранить push-подписку |
| `POST` | `/unsubscribe` | Удалить push-подписку |
| `POST` | `/snooze?reminderId=<id>` | Отложить напоминание на 5 минут |
| `DELETE` | `/reminders/:id` | Отменить запланированное напоминание |

## Проверка функциональности

1. Открыть `https://localhost:3000` и принять самоподписанный сертификат.
2. Разрешить уведомления для сайта, нажав кнопку «Включить уведомления».
3. Создать обычную заметку и проверить мгновенный push.
4. Создать заметку с датой/временем и дождаться отложенного push с кнопкой «Отложить на 5 минут».
5. Нажать «Отложить на 5 минут» и убедиться, что push приходит через 5 минут.
6. Удалить заметку с напоминанием и убедиться, что push не приходит.
7. Открыть вкладку «О приложении» и проверить, что данные сертификата отображаются.
8. Отключить сеть (DevTools → Network → Offline) и убедиться, что интерфейс загружается из кеша.
